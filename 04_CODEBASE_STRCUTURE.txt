# Codebase Structure Blueprint - PROPER 2.9

## Purpose
Define comprehensive file organization, module structure, naming conventions, and code architecture for the PROPER 2.9 AI-Enhanced Hotel Security Platform. This template provides a scalable foundation for organizing the microservices-based security platform.

## Project Root Structure

### Basic Project Information
```
Project Name: PROPER 2.9 AI-Enhanced Hotel Security Platform
Repository URL: https://github.com/proper-security/proper29-platform
Primary Language: Python (Backend Services) + TypeScript (Frontend)
Framework: FastAPI (Backend) + React 18+ (Frontend)
Architecture Pattern: Event-Driven Microservices with CQRS
Team Size: 12-18 developers (4 Frontend, 6 Backend, 3 AI/ML, 2 DevOps, 3 QA)
```

### Root Directory Layout
```
proper29-platform/
├── README.md                           # Project overview and setup guide
├── CHANGELOG.md                        # Version history and security updates
├── LICENSE                             # Enterprise license information
├── .gitignore                          # Git ignore rules for all services
├── .env.sample                         # Environment variables template
├── .env                                # Environment variables (gitignored)
├── docker-compose.yml                  # Local development orchestration
├── docker-compose.prod.yml             # Production container orchestration
├── .github/                            # GitHub Actions CI/CD workflows
├── docs/                               # Comprehensive platform documentation
├── scripts/                            # Build, deployment, and utility scripts
├── services/                           # Microservices backend code
├── frontend/                           # React dashboard application
├── mobile/                             # React Native mobile apps
├── shared/                             # Shared libraries and types
├── tests/                              # Integration and E2E tests
├── infrastructure/                     # Terraform and Kubernetes configs
├── deployment/                         # Deployment configurations
├── monitoring/                         # Monitoring and observability configs
└── data/                               # Database schemas and seed data
```

## Naming Conventions

### File and Directory Naming Patterns
```
General Rules:
- Use kebab-case for directories: security-service, guest-safety-service
- Use snake_case for Python files: incident_service.py, patrol_models.py
- Use camelCase for JavaScript/TypeScript files: securityApi.ts, useAuth.ts
- Use PascalCase for React components: SecurityOverview.tsx, PatrolMap.tsx
- Use descriptive names that indicate purpose and functionality

Specific Patterns:
- Python Models: {entity_name}.models.py (incident.models.py)
- Python Services: {service_name}.service.py (patrol.service.py)
- Python Repositories: {entity_name}.repository.py (user.repository.py)
- Python API Routes: {resource_name}.routes.py (incidents.routes.py)
- TypeScript Types: {domain}.types.ts (security.types.ts)
- React Components: {ComponentName}.tsx (IncidentList.tsx)
- React Hooks: use{HookName}.ts (useRealTimeData.ts)
- API Services: {domain}Api.ts (securityApi.ts)
- Test Files: {component_name}.test.{ext} (incident.service.test.py)
- Configuration Files: {service}.config.{ext} (database.config.ts)
- Constants Files: {category}.constants.{ext} (api.constants.ts)
- Utility Files: {category}.utils.{ext} (validation.utils.ts)
```

### Code Organization Patterns
```
Python Service Module Pattern:
service_name/
├── __init__.py                        # Module exports
├── {service_name}.service.py          # Main service logic
├── models.py                          # Data models
├── repository.py                      # Data access layer
├── schemas.py                         # API schemas
├── exceptions.py                      # Custom exceptions
├── constants.py                       # Service constants
└── utils.py                           # Service utilities

TypeScript Component Pattern:
{ComponentName}/
├── index.ts                           # Component export
├── {ComponentName}.tsx                # Component implementation
├── {ComponentName}.test.tsx           # Component tests
├── {ComponentName}.stories.tsx        # Storybook stories
├── {ComponentName}.module.css         # Component styles
├── types.ts                           # Component types
└── hooks.ts                           # Component hooks

API Route Pattern:
{resource_name}/
├── __init__.py                        # Route exports
├── {resource_name}.routes.py          # Route definitions
├── {resource_name}.schemas.py         # Request/response schemas
├── {resource_name}.dependencies.py    # Route dependencies
└── {resource_name}.middleware.py      # Route middleware
```

1. **Service Development**: Each microservice follows the established patterns with clear separation of concerns
